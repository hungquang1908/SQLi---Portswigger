
# SQL injection examples

- **SQL injection examples**
    
    Có nhiều dạng **SQL Injection (SQLi)** khác nhau, tùy vào cách khai thác:
    
    1. **Lấy dữ liệu ẩn (Retrieving hidden data)**
        - Sửa truy vấn để hiển thị thêm dữ liệu mà bình thường không thấy
    2. **Phá vỡ logic ứng dụng (Subverting application logic)**
        - Thay đổi truy vấn để **làm sai lệch logic** (ví dụ: bypass login)
    3. **UNION attack**
        - Dùng `UNION` để **lấy dữ liệu từ bảng khác** trong database
    4. **Blind SQL Injection**
        - Không thấy kết quả trực tiếp
        - Phải suy đoán dựa vào:
            - Phản hồi đúng/sai
            - Thời gian phản hồi
    
    👉 Nói ngắn gọn:
    
    SQLi có nhiều kiểu khai thác khác nhau, từ đọc dữ liệu, bypass logic đến tấn công “mù” khi không có phản hồi trực tiếp.
    
- **Retrieving hidden data**
    - Ứng dụng web hiển thị sản phẩm theo danh mục, ví dụ:
        
        ```
        /products?category=Gifts
        ```
        
        → Tạo truy vấn SQL:
        
        ```
        SELECT * FROM products WHERE category = 'Gifts' AND released = 1
        ```
        
        - `released = 1` dùng để **ẩn sản phẩm chưa phát hành**
    
    ---
    
    ### ⚠️ Lỗ hổng SQL Injection
    
    - Ứng dụng **không có cơ chế bảo vệ SQLi**
    → Attacker có thể chèn mã SQL vào tham số `category`
    
    ---
    
    ### 1. Bypass điều kiện bằng comment `-`
    
    - Payload:
        
        ```
        Gifts'--
        ```
        
    - Truy vấn thành:
        
        ```
        SELECT * FROM products WHERE category = 'Gifts'--' AND released = 1
        ```
        
    - `-` sẽ **comment phần còn lại**
    → Bỏ điều kiện `released = 1`
    → Hiển thị cả **sản phẩm chưa phát hành**
    
    ---
    
    ### 2. Lấy toàn bộ dữ liệu với `OR 1=1`
    
    - Payload:
        
        ```
        Gifts' OR 1=1--
        ```
        
    - Truy vấn:
        
        ```
        SELECT * FROM products WHERE category = 'Gifts' OR 1=1--' AND released = 1
        ```
        
    - Vì `1=1` luôn đúng
    → Truy vấn trả về **tất cả sản phẩm**
    
    ---
    
    👉 Nói ngắn gọn:
    
    Do không lọc input, attacker có thể chèn SQL để:
    
    - Bỏ qua điều kiện bảo mật
    - Hiển thị toàn bộ dữ liệu trong database
- **Subverting application logic**
    - Ứng dụng đăng nhập kiểm tra bằng truy vấn:
        
        ```
        SELECT * FROM users
        WHERE username = 'wiener' AND password = 'bluecheese'
        ```
        
        → Nếu có kết quả → đăng nhập thành công
        
    
    ---
    
    ### ⚠️ Lỗ hổng SQL Injection (bypass login)
    
    - Attacker có thể chèn payload vào **username**:
        
        ```
        administrator'--
        ```
        
    - Truy vấn trở thành:
        
        ```
        SELECT * FROM users
        WHERE username = 'administrator'--' AND password = ''
        ```
        
    - Dấu `-`:
        - Biến phần còn lại thành **comment**
        - → Bỏ qua kiểm tra password
    
    ---
    
    ### Kết quả
    
    - Hệ thống chỉ kiểm tra:
        
        ```
        username = 'administrator'
        ```
        
    - Không cần password
    → Attacker đăng nhập thành **admin**
    
    ---
    
    👉 Nói ngắn gọn:
    
    SQL Injection có thể dùng để **bypass đăng nhập** bằng cách loại bỏ điều kiện kiểm tra mật khẩu.
    
- **Retrieving data from other database tables**
    - Khi ứng dụng **hiển thị kết quả truy vấn SQL**, attacker có thể khai thác SQLi để **lấy dữ liệu từ bảng khác**.
    - Kỹ thuật sử dụng: **UNION attack**
        - Dùng từ khóa `UNION` để nối thêm một câu lệnh `SELECT` khác
    
    ---
    
    ### Ví dụ:
    
    - Truy vấn gốc:
        
        ```
        SELECT name, description
        FROM products
        WHERE category = 'Gifts'
        ```
        
    - Payload:
        
        ```
        ' UNION SELECT username, password FROM users--
        ```
        
    - Truy vấn sau khi bị chèn:
        
        ```
        SELECT name, description
        FROM products
        WHERE category = ''
        UNION SELECT username, password FROM users--'
        ```
        
    
    ---
    
    ### Kết quả
    
    - Ứng dụng sẽ trả về:
        - **Tên + mô tả sản phẩm**
        - **Username + password từ bảng users**
    
    → Attacker có thể **đánh cắp thông tin đăng nhập**
    
    ---
    
    👉 Nói ngắn gọn:
    
    UNION SQLi cho phép ghép thêm truy vấn để **lấy dữ liệu từ bảng khác**, ví dụ như username và password.
    
- **Blind SQL injection vulnerabilities**
    - **Blind SQL Injection** là trường hợp:
        - Ứng dụng **không trả về kết quả truy vấn**
        - Cũng **không hiển thị lỗi SQL**
        → Việc khai thác **khó hơn**, nhưng vẫn có thể lấy dữ liệu
    
    ---
    
    ### Các kỹ thuật khai thác:
    
    1. **Dựa vào logic đúng/sai (Boolean-based)**
        - Chèn điều kiện đúng/sai vào truy vấn
        - Quan sát sự khác biệt trong phản hồi (nội dung, trạng thái,…)
        - Ví dụ: kiểm tra từng ký tự dữ liệu
    
    ---
    
    1. **Dựa vào thời gian (Time-based)**
        - Tạo câu lệnh gây **delay nếu điều kiện đúng**
        - Nếu phản hồi chậm → suy ra điều kiện đúng
        - Dùng để “đoán” dữ liệu từng bước
    
    ---
    
    1. **Out-of-band (OAST)**
        - Gây ra **kết nối mạng ra ngoài** (DNS, HTTP,…)
        - Có thể:
            - Xác nhận lỗ hổng
            - **Trích xuất dữ liệu trực tiếp** qua kênh này
        - Rất mạnh khi các phương pháp khác không hoạt động
    
    ---
    
    👉 Nói ngắn gọn:
    
    Dù không thấy dữ liệu trực tiếp, Blind SQLi vẫn có thể khai thác bằng cách **suy luận qua phản hồi, thời gian hoặc kênh mạng ngoài**.
    
- **Second-order SQL injection**
    - Có 2 loại **SQL Injection** chính:
    
    ---
    
    ### 1. **First-order SQL Injection**
    
    - Xảy ra ngay khi:
        - Input từ người dùng được đưa trực tiếp vào truy vấn SQL **không an toàn**
    - Lỗ hổng xuất hiện **ngay lập tức** khi gửi request
    
    ---
    
    ### 2. **Second-order SQL Injection (Stored SQLi)**
    
    - Quy trình:
        1. Người dùng nhập dữ liệu → được **lưu vào database** (có thể an toàn lúc này)
        2. Sau đó, ứng dụng lấy dữ liệu này ra
        3. Và **dùng lại trong truy vấn SQL một cách không an toàn**
        → Lúc này mới phát sinh lỗ hổng
    - Nguyên nhân:
        - Developer nghĩ dữ liệu trong DB là **đáng tin cậy**
        - Không kiểm tra lại khi sử dụng → dẫn đến lỗi
    
    ---
    
    ### Đặc điểm
    
    - **First-order**: xảy ra ngay, dễ phát hiện
    - **Second-order**: xảy ra gián tiếp, khó phát hiện hơn
    
    ---
    
    👉 Nói ngắn gọn:
    
    - First-order: lỗi xảy ra ngay khi nhập dữ liệu
    - Second-order: dữ liệu “nằm vùng” trong DB rồi mới gây lỗi khi được sử dụng lại
- **Examining the database**
    - **SQL Injection** có nhiều kỹ thuật hoạt động giống nhau trên các hệ quản trị CSDL (DBMS) khác nhau vì chúng cùng dùng SQL.
    - Tuy nhiên, giữa các hệ CSDL (MySQL, Oracle, SQL Server,…) vẫn có **sự khác biệt**, ví dụ:
        - Cú pháp nối chuỗi
        - Cách viết comment
        - Hỗ trợ query nhiều lệnh (stacked queries)
        - API riêng
        - Thông báo lỗi
    
    → Vì vậy, cách khai thác SQLi có thể **khác nhau tùy hệ CSDL**
    
    ---
    
    ### Sau khi phát hiện SQLi, cần làm gì?
    
    - **Xác định loại database**:
        - Ví dụ với Oracle:
            
            ```
            SELECT * FROM v$version
            ```
            
        - Nếu query chạy được → biết đang dùng Oracle
    
    ---
    
    ### Thu thập thông tin database:
    
    - Liệt kê bảng:
        
        ```
        SELECT * FROM information_schema.tables
        ```
        
    - Sau đó có thể:
        - Tìm các bảng quan trọng (users, accounts,…)
        - Xem các cột (username, password,…)
    
    ---
    
    👉 Nói ngắn gọn:
    
    Mỗi hệ CSDL có đặc điểm riêng, nên sau khi tìm được SQLi, cần xác định loại database và cấu trúc dữ liệu để khai thác hiệu quả hơn.
